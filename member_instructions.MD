## Instructions
- **Assign yourself to a GitHub issue** before starting work.
- Create or contribute to a branch named **`feat/<feature-name>`**, **`fix/<bug-name>`**, or **`docs/<topic>`**.
- Review the project `.md` files and the related issue details before making changes.
- Create new issues for bugs, improvements, or technical debt you discover. Include enough detail to explain the problem and a possible approach to solving it.
- **Communicate with other contributors** working in related areas to avoid duplicated work and merge conflicts.

## Policies
- **Direct pushes to `main` are blocked.** All changes must go through a pull request.
- Keep pull requests **focused, easy to review, and well documented**. Include what changed, why, and how it was tested.
- **AI use is encouraged**, but all generated code must be **reviewed, understood, and cleaned up** before being committed.
- Follow the **existing code style and project structure**.
- Prefer **clean, self-explanatory code** over excessive comments. Use comments only for intent, context, or non-obvious logic.
- **Update documentation, comments, and tests** when relevant.
- **Run tests, linters, and formatters** before submitting changes. (Testing in Jest needs to be fully implemented soon for this purpose)
- Do not commit **debug code, unused imports, or unrelated changes**.
